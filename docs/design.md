# Mobile-first блок покупки в карточке товара — дизайн решения

## Контекст

Тестовое задание: реализовать mobile-first блок покупки в карточке товара на 1С-Битрикс для интернет-магазина детской одежды. Ориентир — 6–8 часов.

## Стек

- **Платформа:** 1С-Битрикс «Малый бизнес» (с модулем `catalog` и торговыми предложениями)
- **Развёртывание:** Docker (PHP 8.1 + Apache + MySQL 8.0)
- **Frontend:** ES6 classes, vanilla JS, без фреймворков
- **Шаблон:** кастомный шаблон компонента `bitrix:catalog.element` в `/local/templates/`

## Структура карточки (mobile-first)

Первый экран сверху вниз:

1. Галерея (свайп-слайдер, точки-индикаторы)
2. Название товара + бренд
3. Цена + старая цена
4. Наличие (бейдж «В наличии» / «Нет в наличии»)
5. Блок выбора SKU:
   - Цвет — круглые свотчи
   - Размер — чипы (кнопки)
6. Кнопка «В корзину»

Sticky-bar снизу экрана при скролле: миниатюра + цена + кнопка.

## Данные

### Инфоблок «Товары»

| Свойство | Тип | Назначение |
|----------|-----|-----------|
| CML2_ARTICLE | string | Артикул |
| MORE_PHOTO | file (multiple) | Галерея |

### Инфоблок «Торговые предложения» (SKU)

| Свойство | Тип | Назначение |
|----------|-----|-----------|
| CML2_LINK | element → Товары | Связь с товаром |
| COLOR_REF | list | Цвет (список значений с HEX) |
| SIZE_REF | list | Размер (список значений) |

Цена — через модуль catalog (`CPrice`).
Наличие — через `CATALOG_QUANTITY`.

## Тестовые данные

2 товара:
- Футболка «Мишка» — 3 цвета × 3 размера = 9 SKU (часть нет в наличии)
- Платье «Ромашка» — 2 цвета × 4 размера = 8 SKU

Данные наполняются через `seeder.php` либо восстанавливаются из MySQL-дампа в `docker/db/dump.sql.gz` (для воспроизводимости у ревьюера).

### Разделение ответственности

- **result_modifier.php** — выборки, группировка SKU в матрицу `{color → {size → offer}}`, вычисление доступности комбинаций
- **template.php** — только HTML, без бизнес-логики
- **style.css** — mobile-first, tap-зоны ≥44px, без hover
- **script.js** — класс `ProductCard`: `selectColor()`, `selectSize()`, `updateView()`, `addToCart()`
- **add_to_cart.php** — принимает `offer_id` + `quantity`, использует `Bitrix\Sale\Basket`, возвращает JSON

## Поведение SKU-селектора

1. При загрузке выбран первый доступный цвет + первый доступный размер
2. Клик по свотчу цвета:
   - Обновляются доступные размеры (недоступные — `disabled`, но видны)
   - Если текущий размер недоступен — выбирается первый доступный
   - Обновляются цена, наличие, галерея (если у SKU своё фото)
3. Клик по чипу размера — аналогично для размеров
4. Все данные SKU уже на странице в JSON (inline `<script type="application/json">`), никаких AJAX при переключении

## AJAX-корзина

**Endpoint:** `POST /local/ajax/add_to_cart.php`

**Body:** `{ offer_id: number, quantity: 1 }`

**Логика:**
1. CSRF-проверка через `bitrix_sessid`
2. `Basket::loadItemsForFUser(Fuser::getId(), SITE_ID)` → `getExistsItem()` или `createItem()`
3. `setFields(['QUANTITY' => ..., 'PRICE' => ..., 'PRODUCT_PROVIDER_CLASS' => ...])`
4. `basket->save()`
5. Ответ: `{ status: "ok", cart_count: N, cart_total: "..." }`

**Frontend:**
- Клик → кнопка `disabled` + спиннер (защита от повторного клика)
- Успех → кнопка `«Добавлено ✓»` на 1.5 сек → возврат в `«В корзине (N)»`
- Ошибка → `«Ошибка, попробуйте ещё»`

## Sticky purchase bar

- Появляется при скролле ниже основной кнопки (IntersectionObserver)
- Содержит: название товара (1 строка, ellipsis), цена, кнопка «В корзину»
- На десктопе скрыт (`@media (min-width: 768px)`)
- Кнопка в sticky-bar вызывает тот же `ProductCard.addToCart()`

## Воспроизводимость

Ревьюер делает:
```bash
git clone ...
cd Sofuto_bitrix
docker compose up -d
# Дамп БД восстанавливается автоматически через /docker-entrypoint-initdb.d/
open http://localhost:8080/catalog/
```

Админка: `admin / admin123` (задаётся в дампе).

## Что не входит в scope

- Хлебные крошки, рейтинг, отзывы, похожие товары
- Табы «Описание/Характеристики»
- Зум галереи, лайтбокс
- WebP-конвертация, lazy-loading (покажем только `loading="lazy"` атрибут)
- Десктоп-вёрстка (минимальная адаптация через брейкпоинт)
- «Сообщить о поступлении»
- Композитный сайт, настройка кэша
- Оптимизация Aspro (пилот без Aspro вообще)

## Риски

1. **Установка «Малый бизнес» тяжёлая** — если зависнет, переходим на «Бизнес»
2. **API торговых предложений** — `CCatalogSku::getOffersList()` имеет нюансы; используем проверенный шаблон с `Bitrix\Iblock\ElementTable` + `Bitrix\Catalog\PriceTable`
3. **Дамп БД большой (~50 МБ)** — кладём в `docker/db/dump.sql.gz` (gzip, ~5-10 МБ)
4. **Время 6-8 часов** — жёсткий бюджет; при угрозе просрочки срезаем sticky-bar и галерею-свайп (заменяем на одно фото)
